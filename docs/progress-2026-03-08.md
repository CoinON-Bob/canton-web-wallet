本文档为当前阶段的成果备份说明：已完成功能、已打通流程、仍为 Mock 的部分、真实 Canton 所需外部条件、客户需提供信息及下一步计划。

## 一、今天完成了哪些功能

1. **真实 Canton 接入骨架**
   - 后端增加 `CANTON_USE_SDK` 开关与 `KEY_ENCRYPTION_KEY`、`CANTON_DEFAULT_INSTRUMENT` 等环境变量（`api/src/config/env.schema.ts`）。
   - 实现 `CantonService`：连接 SDK、分配 external party、暴露 `getSdk()` 供钱包/Token Standard 使用。
   - 实现钱包私钥加密存储：`WalletKeystoresService` 的 `setCantonPrivateKey` / `getCantonPrivateKey`，使用 AES-256-GCM（`KEY_ENCRYPTION_KEY`）。

2. **真实 Canton 钱包流程**
   - `CantonWalletService`：创建钱包 = 分配 party + 加密存私钥；余额 = Token Standard `listHoldingUtxos` 汇总；转账 = `createTransfer` + `prepareSignAndExecuteTransaction` 用存储私钥签名提交。
   - `WalletsService` 在 `CANTON_USE_SDK=true` 时，创建/列表/单查/转账全部委托给 `CantonWalletService`；否则继续走 Mock Node（`NODE_API_BASE_URL`）。

3. **文档**
   - `docs/real-canton-setup.md`：如何切真实 Canton、环境变量、LocalNet 与 NaaS 说明、**有无官方/公共节点**的结论、与客户/运维的沟通清单。
   - 明确：官方无免申请公共节点；第三方如 cantonnodes.com 仅提供 Scan（只读）API，不能替代 Ledger API 做创建钱包与转账。

4. **测试与构建**
   - Mock 模式下 E2E 通过：注册 → 验证邮箱 → 登录 → 创建两个钱包 → 转账 → 刷新余额。
   - API 构建通过。

## 三、当前哪些仍然是 Mock

- **默认运行模式**：未设置 `CANTON_USE_SDK=true` 时，钱包创建/余额/转账全部走 **Mock Node**（`api/scripts/mock-node-api.js`），即内存账户，无真实链。
- **邮件**：邮箱验证仍为 Mock（如控制台输出验证码或本地 mock 服务），未接真实 SMTP/邮件服务。
- **Canton 连接目标**：真实模式下，SDK 连接逻辑目前写死为 **LocalNet 默认**（localhost Ledger API + Scan Proxy）；尚未从环境变量读取 NaaS/客户提供的 Ledger URL、Topology URL 及认证。

---

## 四、真实 Canton 还缺哪些外部条件

1. **可用的 Ledger API 端点**
   - 要么本地起 **Canton LocalNet**（Docker Compose），并保证 Ledger API（如 2901）与 Scan Proxy 可访问；
   - 要么由 **NaaS 或客户** 提供 Ledger API 基址（gRPC）及认证方式。

2. **认证与网络**
   - 若为 NaaS/客户节点：需提供 OAuth2 / JWT / API Key 等认证信息（client_id、client_secret、token URL、scope/audience 等，视对方要求）；
   - 若为 DevNet/TestNet：通常需固定出口 IP、OIDC 等，按官方 Validator Onboarding 流程申请。

3. **默认 Instrument**
   - 若环境使用的代币不是 Amulet，需确认并配置 `CANTON_DEFAULT_INSTRUMENT`。

4. **策略与限制**
   - 确认是否允许本应用在该节点上创建 external party（钱包），以及 IP/速率限制。

## 五、客户需要提供哪些 API / 认证信息

与客户/运维沟通时，可直接引用 `docs/real-canton-setup.md` 中「五、与客户/运维沟通清单」。摘要如下：

| 项目 | 说明 |
|------|------|
| **Ledger API 基址** | 例如 `https://...`，SDK 连接 Participant 提交命令与查询。 |
| **Scan Proxy / Topology API 基址** | 若有独立 Topology 服务，需一并提供。 |
| **认证方式** | OAuth2 / JWT / API Key；若 OAuth2，需 client_id、client_secret、token URL、scope/audience（若适用）。 |
| **当前网络** | DevNet / TestNet / MainNet。 |
| **默认代币 instrument id** | 若与 Amulet 不同，需告知。 |
| **是否允许创建 external party** | 以及 IP/速率限制。 |

拿到上述信息后，可在 `api/src/canton/canton.service.ts` 中接入环境变量（如 `CANTON_LEDGER_URL`、`CANTON_TOPOLOGY_URL`）并替换认证逻辑（如 `ClientCredentialOAuthController`）。

## 六、下一步开发计划

1. **等客户/运维提供**：Ledger API 与 Topology 基址、认证方式、网络与 instrument 信息；或自行部署 LocalNet 做联调。
2. **配置扩展**：在 `canton.service.ts` 中支持通过环境变量配置 Ledger URL、Topology URL 及 OAuth2/JWT，替代当前 LocalNet 写死逻辑。
3. **真实环境自测**：在 LocalNet 或 NaaS 配置完成后，完整跑一遍：注册 → 验证 → 登录 → 创建两个钱包 → 转账 → 刷新余额，确认链上一致。
4. **（可选）** 若仅需只读验证链上数据，可后续增加对第三方 Scan API（如 cantonnodes.com）的只读客户端，用于查询 holdings 等；不影响当前「创建钱包 + 转账」的 Ledger API 路径。
